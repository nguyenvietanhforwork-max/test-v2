# Repository Audit — Atlas / Value Research Vault

This audit is Phase 1 of the restructure described in `restructure backend.md`. It classifies every top-level path as **canonical**, **duplicate**, **experimental**, **legacy**, **scaffolding**, or **content**, and surfaces the real architectural forks that need a human decision before any file moves.

## 1. The duplication is real

The prompt's diagnosis was correct. This repository contains at least four overlapping layers of the same product idea, built at different times, none of them fully cleaned up:

1. **L1 — Obsidian vault (working, with real content):** `raw/`, `wiki/`, `Processed/`, `_templates/`, `index.html`, `news_database.json`, and three root-level Python scripts (`process_news.py`, `process_news_agent.py`, `generate.py`). Governed by `AGENTS.md` v2.0 and `CLAUDE.md`. Contains ~62 real news markdown files, ~35 wiki notes, and 4 generated reports dated 2026-05-14.

2. **L2 — "Atlas v2" full-stack monorepo:** `apps/web/` (Next.js 15 + React 19, App Router, real components), `apps/api/` (FastAPI 0.115 + SQLAlchemy 2 + alembic), `packages/agents/`, `packages/automation/`, `packages/shared/`, `infra/docker/`, `infra/k8s/`, `demo/dashboard.html`, `docs/`. Governed by `README.md` and `ARCHITECTURE.md`. Real code, untested whether it currently runs end-to-end.

3. **L3 — `intelligence-platform/` (another parallel monorepo):** Its own `apps/api/`, `apps/dashboard/`, `services/{agents,ingestion,workers}`, `n8n/workflows/`, `obsidian/plugin/`, `infra/supabase/`, `infra/railway/`, `infra/docker/`, plus its own `pnpm-workspace.yaml`, `.env`, `.github/workflows/`. Overlaps semantically with L2 in almost every directory.

4. **L4 — Loose experimental dashboards and prompt artifacts at root:** `news-dashboard-v2.html`, `news-dashboard-soulslab.html`, `offline dashboard trial/` (5 HTMLs), plus `ai_intelligence_prompt_bundle.zip` and five "prompt" markdown files (`production_ai_intelligence_platform_prompt.md`, `infrastructure_prompt.md`, `dashboard_design_prompt.md`, `ai_dashboard_redesign_prompt.md`, `agent.md`).

## 3. The hard architectural forks (need Albert's call)

This is the heart of the restructure. None of these can be answered safely without you:

### Fork A — Canonical content/data plane

- **Option A1:** Keep L1 vault layout (`raw/` immutable, `wiki/` AI-owned, `Processed/` reports, `_templates/`). This is what `AGENTS.md` is built around and what the actual content lives in. The new prompt's `content/`, `cleaned/`, `generated/` directories are added *alongside*, not as replacements.
- **Option A2:** Migrate to the prompt's target layout (`raw/`, `cleaned/`, `content/`, `generated/`, `feedback/`, `prompts/`, `scripts/`, `schemas/`). This requires moving every note from `wiki/` into `content/` and updating every `[[wikilink]]` and every frontmatter `sources:` reference.

**Recommendation:** A1 + additive. The prompt's target is mostly compatible: add `cleaned/`, `generated/`, `feedback/`, `prompts/`, `scripts/`, `schemas/` to the root, but keep `raw/`, `wiki/`, `Processed/` exactly as `AGENTS.md` defines them. Treat `wiki/` as the prompt's "content/" by convention, and add a `content/` symlink-equivalent (a README that points to wiki/) if needed.

### Fork B — Canonical dashboard

Live candidates: `apps/web/` (Next.js, real code), `index.html` (single-file, working), `demo/dashboard.html`, `intelligence-platform/apps/dashboard/`, plus three generated dashboards in `outputs/` and seven HTMLs in `news-dashboard-*` / `offline dashboard trial/`.

**Recommendation:** Pick one of two coherent paths:

- **B1 — "Markdown-driven simple":** Canonical dashboard is the current `index.html` (with future iteration). It reads `news_database.json` and the markdown files directly. No build step, no Next.js, no Node. This matches the prompt's "simplicity first" and "remove giant static HTML architecture" *if* we rewrite `index.html` to be markdown-driven instead of JSON-bound. All Next.js + Atlas v2 frontend goes to `legacy/`.
- **B2 — "Keep apps/web":** Canonical dashboard is `apps/web/`. Continue iterating in Next.js. Then `index.html` becomes a legacy single-file fallback (already explicitly framed that way in `README.md` §6). All other HTMLs to `legacy/`.

Both are reasonable. B1 is closer to the new prompt's spirit; B2 preserves the most engineering work.

### Fork C — Canonical backend

- `apps/api/` (FastAPI, real code) vs. `intelligence-platform/apps/api/` (parallel FastAPI). Pick one. The prompt explicitly says "Keep FastAPI."

**Recommendation:** `apps/api/`. Archive `intelligence-platform/` whole. The prompt also says to strip out unnecessary abstraction, fake enterprise patterns, premature scalability — but that is post-archive cleanup, not a fork.

### Fork D — Canonical pipeline

- `packages/automation/` (LangGraph + Celery + watchdog) vs. `intelligence-platform/services/{ingestion,workers,agents}` vs. the L1 Python entry points (`process_news.py`, `process_news_agent.py`, `generate.py`).

**Recommendation:** `packages/automation/` is the most developed. Promote it. Archive `intelligence-platform/services/`. **Read `process_news.py`/`process_news_agent.py`/`generate.py` before archiving** — `AGENTS.md` §16 calls them out by name as the L1 orchestrator stubs, so they may still be the entry point you actually run by hand.

### Fork E — Canonical deployment

Four configs at root (`docker-compose.yml`, `vercel.json`, `fly.toml`, `render.yaml`) plus `intelligence-platform/infra/{docker,railway,supabase}` plus `apps/api/railway.json` plus `apps/api/Procfile` plus `infra/k8s/`.

**Recommendation:** Pick **one dev stack** (`docker-compose.yml`) + **one prod target**. The prompt forbids Kubernetes, so `infra/k8s/` → legacy. Among `vercel.json` / `fly.toml` / `render.yaml` / Railway, pick the single one you actually deploy to and archive the others.

### Fork F — Legacy preservation depth

`intelligence-platform/` contains its own Supabase migrations (`0001`–`0004`) and n8n workflows (`daily-report`, `ingest-news`, `weekly-wiki-lint`) and an Obsidian plugin source. Are any of those wired up to your current vault? If yes, they need to be promoted out of `intelligence-platform/` before the rest of it goes to `legacy/`.
